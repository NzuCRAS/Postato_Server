# 结构树图谱化重构方案(依赖图 + 模块档案)

> 状态:设计待审。决策已定:扁平模块+单层 group / 废弃重建 / 手动声明依赖边 / 先出文档再做。
> (本文档为设计稿,审定后同步落知识库 /design/ 并按 §8 立需求实施。)

## 1. 背景与问题
现状:结构树是 DDD 分类树(L0 系统→L1 域→L2 上下文→L3 模块),约 40 节点。三大问题:
1. **中间分类层(域/上下文)是纯容器**,不承载实质信息——「知识域/知识库上下文」对人和 LLM 都是噪音。
2. **树只表达包含/归属,不表达依赖/调用**——丢失了「谁调谁、数据怎么流」这一理解架构的核心。
3. **节点缺索引**——只有 impl_status + related_code,没把模块和它的需求/技术决策/经验连起来。

## 2. 目标
- 人能一眼看懂:模块 + 依赖关系。
- LLM 一次读到架构全貌 + 任一模块的完整上下文。
- 每模块可追溯其需求 / 技术决策 / 经验。

## 3. 核心模型:依赖图 + 模块档案
### 3.1 模块节点 ArchModule(唯一节点实体)
- `key`:稳定标识(如 `permission`、`mcp-gateway`),引用/关联都用它
- `title`、`description`
- `group`:粗分组/子系统名(如「用户与权限」「知识库」「集成层」)——**仅用于 mermaid subgraph 视觉聚类,不是节点、不承载关联**
- `impl_status`:planned | in_progress | done(模块自身状态;无严格父子,不再自下而上聚合,可选按 group 汇总展示)
- `related_code`:glob

### 3.2 依赖边 ArchEdge(新增一等公民)
- `from` → `to`(模块 key)
- `kind`:calls | depends | data_flow | auth …
- `label`:可选说明(如「JWT 校验」)
- 手动声明(MCP 工具),业务依赖图由人定,不被代码细节污染。允许环(双向依赖),展示按 kind 着色。

### 3.3 模块档案(索引,核心诉求)
每模块挂三类关联,每条带 `scope`(涉及模块 key 列表):
- `requirements`:需求文档(复用现有 relate_requirement_arch)
- `tech_docs`:技术说明(中间件选用/重要决策 → tech-proposal / standard 页)
- `experiences`:经验文档(experience 页)

**跨模块文档**(如网关↔订单的鉴权需求):关联仍挂模块,`scope=[gateway, order]` → **两个模块档案里都出现,且标注「跨 gateway↔order」**。满足「两端都建索引 + 说明跨哪些模块」,又避免边/节点两套索引。边只管依赖拓扑。

## 4. 展示
- **总览页**:一张 mermaid `flowchart`——模块=节点、依赖=边、group=`subgraph` 聚类;喂给现成 `MermaidBlock` 渲染。点节点 → 模块档案。
  - 生成示例:`flowchart TD; subgraph 知识库; wiki; asset; end; mcp-->|调用| wiki`
- **模块档案页**:入边/出边(依赖谁、被谁依赖)+ 需求/技术/经验索引(跨模块标 scope)+ related_code + impl_status。

## 5. 给 LLM 用(MCP)
新增 resource:`potato://arch/overview`(整张依赖图 mermaid)+ `potato://arch/<key>`(模块档案 markdown)——LLM 一次读到架构全貌 + 模块上下文。这才是真正「告诉 LLM 项目架构」。

## 6. 迁移(废弃重建)
- 现有 arch_nodes 整体归档(status=archived 或迁旧集合,不物理删,留追溯)。
- 重新声明模块 + group + 依赖边。
- 现有 relate_requirement_arch 关联(指向旧 L3 模块 path)手动重挂到新模块 key(量不大)。

## 7. MCP 工具变更
- `upsert_arch_layer`(逐层建树)→ 替换为 `upsert_arch_module`(模块)+ `upsert_arch_edge`(依赖边)。
- `relate_requirement_arch` → 扩展为 `relate_arch_doc`:支持挂 requirement / tech_doc / experience + scope。
- `get_architecture` → 返回 模块 + 边(或直接生成 mermaid + 档案 JSON)。
- 新 arch resource(见 §5)。

## 8. 实施分解(后续按 potato 立需求)
1. **后端模型**:ArchModule + ArchEdge 实体/仓储/服务/CRUD + 旧树归档。
2. **MCP 工具**:upsert_module/edge、relate_arch_doc、get_architecture 改造、arch resource。
3. **前端**:依赖图总览(mermaid)+ 模块档案页。
4. **数据重建**:重新声明 Potato 平台自身的模块图 + 边 + 重挂需求/技术/经验索引。

## 9. 待决 / 风险
- impl_status 不再聚合(图无严格父子);若想要「整体完成度」可按 group 或全模块比例展示。
- 依赖边可能成环(双向依赖)——mermaid 支持,设计上接受并按 kind 区分。
- scope 关联一致性:删模块时需清理/警示引用它的 scope(类似权限字典删除保护)。
- group 单层够不够:超大项目可能要两级;先单层,group 字段保留点分隔扩展余地(`集成层/MCP`)。
