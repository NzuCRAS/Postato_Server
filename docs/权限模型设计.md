扁平团队用**纯职能标签**是最干净的方案，砍掉职级后模型更简单，而 MongoDB 的文档模型天然适合存储这种“用户标签数组”和“权限配置”，扩展起来毫无 schema 变更压力。

---

## 最终权限模型：纯职能驱动（MongoDB 原生适配）

### 1. 用户集合文档
```javascript
// users 集合
{
  _id: ObjectId,
  username: "boss",
  passwordHash: "...",
  functions: ["product"],          // 职能标签数组，可多个
  apiKey: "mcp_live_xxxx",        // 可选，开发者才有
  createdAt: ISODate,
  updatedAt: ISODate
}
```
- `functions` 可选值：`admin`（平台管理）、`product`（产品）、`development`（开发）、`testing`（测试）。
- 老板给 `["product"]`，开发给 `["development"]`，你给自己 `["admin", "development"]`。
- 同一个人可以身兼多职（如 `["development", "testing"]`）。

### 2. 权限配置集合（分离出权限规则）
为了未来能动态调整，可以把权限规则也存进 MongoDB，启动时加载到内存缓存。

```javascript
// permission_rules 集合
{
  _id: ObjectId,
  resource: "requirement",
  action: "edit_structured",
  description: "编辑需求结构化内容",
  requiredFunctions: ["product", "admin"]   // 允许的职能列表
}
```
- 判断时：如果用户的 `functions` 与规则的 `requiredFunctions` 有交集，即通过。
- 超级管理员豁免：只要用户拥有 `admin` 标签，直接全部权限。（也可以在规则中不写 admin，单独判断）

### 3. 权限判断逻辑
```javascript
function hasPermission(user, resource, action) {
  // 1. admin 豁免
  if (user.functions.includes('admin')) return true;
  
  // 2. 从缓存中查找规则
  const rule = getRule(resource, action); // 从 permission_rules 加载的 Map 中取
  if (!rule) return false; // 默认禁止
  
  // 3. 检查职能交集
  return user.functions.some(f => rule.requiredFunctions.includes(f));
}
```

### 4. 如何新增职能或调整权限
- **新增职能**：在 `permission_rules` 中插入新规则或在现有规则的 `requiredFunctions` 中添加新标签即可，无需改代码（除非需要特定的业务逻辑，但初期不需要）。
- **调整某职能的权限**：直接修改对应规则的 `requiredFunctions` 数组，重启或动态刷新缓存后即刻生效。

**MongoDB 的优势**：文档模型允许随时嵌套复杂结构，未来如果规则变复杂（比如增加数据范围限制），直接在规则文档里加字段，不影响已有代码。
