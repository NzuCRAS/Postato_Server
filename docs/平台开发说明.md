## 1. 项目说明

Potato 是**团队内部研发效能平台**,三大诉求:**需求对齐 + 知识沉淀 + vibecoding(AI 辅助编程)**。

**核心假设(平台的灵魂):** 通过「结构化需求 + Herness 进度树 + MCP 契约」,让 Claude Code **步步为营、可追溯、可干预**地开发,而不是一步把代码生成出来。这是它区别于普通项目管理工具的根本。

**五大模块:**
| 代号 | 模块 | 说明 |
|------|------|------|
| A | 需求澄清与结构化 | 模糊描述 → 结构化需求(单一事实来源),`draft→clarifying→confirmed` |
| C | Herness 进度树 | 与需求绑定、AI 生成/人可干预的「模块分解+进度追踪树」(非甘特图) |
| D | 知识库 Wiki | 树状文档,供人查阅 + AI 检索 |
| E | 工单系统 | 轻量 Bug/任务,关联需求与进度节点 |
| — | 权限 | 纯职能驱动(admin/product/development/testing) |
| — | MCP Server | 把平台能力封装成 Claude Code 可调用的 Tool |

### 2.3 关键设计决策(原始文档没有的部分,重点 ⭐)

> 这些是本会话与用户逐条对齐、原始 6 份设计文档**未涵盖或需要修正**的决策。接手者必须知道。

1. **技术栈拍板**:后端用 **Java/Spring Boot**(用户个人偏好),**否决了**「统一 TypeScript 全栈」方案。MCP 独立为 TS 服务。
2. **统一认证 + 权限单点(修正原文档矛盾)**:
   - 原 MCP 文档说「MCP 启动时把 apiKeys 加载进内存」,却又说「MCP 不碰 DB」——**自相矛盾**。
   - **现行方案**:后端认证层**同时支持 JWT(Web)与 API Key(MCP)**;MCP Server 只做协议转换 + **透传 `Authorization` 头**,自身**无鉴权/权限逻辑**。
   - **权限判定 `hasPermission` 全部在后端**,避免 Java/TS 两份规则。
3. **Project→Repo 两层结构(原文档只有悬空的 `projectId`)**:识别出「项目 → 子代码仓库」两层;**第一切片只用 1 项目 + 1 仓库默认实例、无管理 UI**,各实体带 `projectId` + 可选 `repoId` 预留,未来扩展不需迁移数据模型。
4. **`structured` 与 JSON 字段命名**:`structured`(对 MCP/AI 暴露的领域契约)内部字段用 **snake_case**(`user_stories`/`acceptance_criteria`…),与设计文档/MCP 契约一致;**平台外壳 API 用 camelCase**(`userId`/`updatedAt`…)。后端用 `@JsonProperty` 控制。
5. **默认管理员 seed**:启动时若无 `admin` 用户,自动创建 **`admin` / `admin123`**(可用环境变量 `SEED_ADMIN_PASSWORD` 覆盖,生产必改)。
6. **权限规则 seed**:启动时若 `permission_rules` 为空,写入初始规则(见 §6)。`PermissionService` 当前**直查库、未加缓存**(plan 规划了缓存,后续优化)。
7. **API Key 格式**:`mcp_live_<32位hex>`,明文仅在生成时返回一次,列表只显示脱敏前缀。
8. **开发期全容器化**:`docker compose` dev 模式——官方镜像(`maven`/`node:20`/`mongo:7`)+ 挂载源码 + 热重载。服务间用**服务名**通信(`mongo`/`backend`),地址由环境变量注入,本机直跑回退 `localhost`。
9. **部署方向**:Dockerfile 多阶段镜像 + docker-compose + GitHub Actions(**留到「部署切片」**,当前未做)。
10. **状态机**:目前仅校验状态值合法(`draft|clarifying|confirmed|deprecated`),**未强制流转方向**(后续加严)。
11. **编辑器简化**:`structured` 用「JSON 文本框 + 提交前校验」,**未引入 monaco/md-editor**(后续切片升级)。
12. **推进方式**:**纵向切片**。第一切片 =「灵魂闭环」(MCP 驱动进度树)。
13. **Herness 进度树做深(2026-06**:把进度树做成「信息完备、可验收、可追溯、可干预」的驾驭树。要点:
    - **GitHub 闭环**:`DevPlan` 顶层记 `repo`;**commit 挂在节点工作日志条目上**(一节点 N 条日志 N 个 commit);`repo.url + sha` 拼出可点的 commit 链接。回填靠人工/MCP 主动上报,**不接 GitHub API/webhook**。
    - **节点信息深度(A)**:`module_ref` 继承需求模块验收标准 + 节点自带可勾选 `acceptance_criteria`;`related_docs` 关联知识库;结构化 `artifacts`。
    - **可观测可干预(C)**:前端树为主 + 顶部总览条(状态计数/阻塞高亮/仓库链接)+ 节点详情抽屉;人对 AI 的自然语言纠偏(`corrections`,AI 经 MCP 读到"未解决纠偏")。
    - **过程契约(B,轻量)**:硬规则仅"blocked 必填原因";其余为软警告(done 无产物/验收未勾/子节点未完成);状态变更自动追加日志;**`actor` 由认证渠道推断**(API Key→ai,JWT→human,见 `AuthTokenFilter` 注入的 `CHANNEL_*` authority)。
    - 设计/计划文档:`docs/superpowers/specs/2026-06-09-herness-dev-plan-deepening-design.md`、`docs/superpowers/plans/2026-06-09-herness-dev-plan-deepening.md`。
    - 暂缓(D 轮):`write_tech_proposal`、契约 prompt 重写、GitHub webhook、依赖建模。

## 5. MCP Server 设计(🔲 里程碑 5/6 实现)

独立 TS 服务,Streamable HTTP（`POST /mcp`)。当前仅占位 `ping` 工具。计划 4 个 Tool:

| Tool | 后端 API | 权限职能 |
|------|----------|----------|
| `get_requirement_detail` | `GET /requirements/{id}` | development/testing/product |
| `search_knowledge` | `GET /wiki/search` | 所有认证用户 |
| `create_dev_plan` | `POST /requirements/{id}/dev-plan` | development |
| `update_dev_plan_node` | `PATCH /requirements/{id}/dev-plan/nodes/{nodeId}` | development |

**鉴权:** MCP 从 `Authorization: Bearer mcp_xxx` 取 Key,**原样透传**给后端;后端验 Key + 判权限。MCP 自身不判权限。

> **进度树做深后(2026-06)**:`create_dev_plan` 入参支持 `repo` 与节点 `module_ref/acceptance_criteria/related_docs`;`update_dev_plan_node` 支持 `commit/log_detail` 并透传后端 `warnings`(软警告);`get_requirement_detail` 的 dev_plan 摘要带上 `acceptance_criteria`(含 checked)、`related_docs`、未解决 `corrections`、`repo`。

> **D 轮(2026-06)**:MCP 工具增至 **9** —— 新增 `add_dev_plan_nodes`(加子节点)、`reset_dev_plan`(重置入档)、`set_dev_plan_repo`(改仓库)、`write_knowledge`(知识沉淀:按 path upsert wiki 页)、`write_tech_proposal`(文档先行:建技术方案 wiki 页+关联节点+可选翻 in_progress)。技术方案/沉淀知识**复用 Wiki**(路径+标签区分临时与 durable),`wiki/edit` 权限扩到 `development`。Herness 契约见仓库根 `CLAUDE.md` 与知识库 `/agent/herness-contract`。

**Herness 契约 prompt**(里程碑 5/6 时写入 Claude Code 配置):AI 接到任务须依次 `get_requirement_detail → search_knowledge → create_dev_plan → 逐节点 update_dev_plan_node`,详见原始 `mcp server设计.md` 第 7 节。

## 6. 权限模型

- 纯职能驱动:`admin` / `product` / `development` / `testing`,用户可多职能。
- `hasPermission(user, resource, action)`:**admin 豁免一切**;否则查 `(resource,action)` 规则,用户职能与 `requiredFunctions` 有交集即通过;无规则默认拒绝。
- 后端 `PermissionService.check()` 在 Controller 里调用,无权限抛 403。

**初始 seed 规则(8 条):**
| resource | action | requiredFunctions |
|----------|--------|-------------------|
| requirement | view | development, testing, product |
| requirement | create / edit_structured / update_status | product |
| wiki | read | development, testing, product |
| wiki | edit | product |
| dev_plan | create / update | development |

## 7. 代码组织与约定

### 7.1 仓库结构
```
Potato Server/
├── backend/   Spring Boot:com.potato.{ , health, auth, user, permission, requirement, config }
├── frontend/  React+Vite:src/{ api, auth, features, pages, components, types }
├── mcp-server/ TS:src/{ index.ts, (tools/ 待建) }
├── docs/      设计文档(本文件 + 原始 6 份)
└── docker-compose.yml
```

### 7.2 后端约定
- 实体用 lombok `@Data` + `@Document`;Repository 继承 `MongoRepository`。
- 业务逻辑放 `XxxService`,HTTP + 权限检查放 `XxxController`,DTO 用 `record`。
- 认证:`AuthTokenFilter`(OncePerRequestFilter)解析 JWT 或 API Key,把 `User` 放进 SecurityContext;`@AuthenticationPrincipal User` 取当前用户。

### 7.3 前端约定 ⭐(用户明确要求:视图与逻辑分离)
| 层 | 目录 | 职责 |
|----|------|------|
| 数据访问 | `src/api/` | `client.ts` 封装 HTTP/注入 token;按域分 `auth.ts`/`user.ts`/`requirement.ts`。纯函数 |
| 逻辑 | `src/auth/AuthContext.tsx`、`src/features/useXxx.ts` | 状态、副作用、业务编排,用自定义 hook 暴露 |
| 视图 | `src/pages/`、`src/components/` | 只渲染 + 把事件转给 hook,**绝不**直接 `fetch` 或写业务逻辑 |
| 类型 | `src/types/` | 共享类型 |

## 8. 本地开发与验证指南

**前置:** Docker(开发期不需要本机装 JDK/Maven/Node)。

```bash
docker compose up -d                 # 起全部:mongo + backend + frontend + mcp(首次较慢,装依赖)
docker compose logs -f backend       # 看 "Started PotatoApplication" 即就绪
```

| 服务 | 地址 |
|------|------|
| 前端 | http://localhost:5173 |
| 后端健康 | http://localhost:8080/health |
| MCP 健康 | http://localhost:3001/health |

**默认账号:** `admin` / `admin123`

**开发循环:**
- 改**前端**代码 → Vite 热重载,浏览器刷新即可(新增依赖才需 `docker compose restart frontend`)。
- 改**后端** Java → `docker compose restart backend`(容器内 `mvn spring-boot:run` 重新编译;约 1~3 分钟)。
- 改 **MCP** → `docker compose restart mcp-server`。

**冒烟(PowerShell):**
```powershell
$t = (Invoke-RestMethod http://localhost:8080/api/v1/auth/login -Method Post -ContentType 'application/json' -Body '{"username":"admin","password":"admin123"}').token
Invoke-RestMethod http://localhost:8080/api/v1/requirements -Headers @{Authorization="Bearer $t"}
```

## 12. 给接手者(含 AI 会话)的继续开发指引

**当前停在:** 里程碑 4 完成(知识库),MCP 4 个工具已实现;**Herness 进度树做深已落地**(节点信息深度 A + 可观测可干预 C + 轻量契约 B + GitHub commit 闭环;后端编译/单测通过、前端 tsc 通过,详见 §2.3 第 13 条)。下一步:**前端界面统一验证 + 里程碑 7 用 Claude Code 端到端联调**。

**MCP 实现要点(里程碑 5/6,`mcp-server/`):**
1. 无状态 Streamable HTTP:每个 `/mcp` 请求从 `Authorization` 头取 API Key → 注入 `buildServer(apiKey)` → tool handler 闭包透传给后端(`api-client.ts`)。MCP 自身不判权限,后端按 API Key 对应职能判定。
2. 4 个工具:`get_requirement_detail`(GET /requirements/{id} → structured + dev_plan 摘要)、`search_knowledge`(GET /wiki/search)、`create_dev_plan`(POST .../dev-plan)、`update_dev_plan_node`(PATCH .../nodes/{id})。
3. **MCP 无前端界面**:`restart mcp-server` 看日志只验证「服务起来」;**工具的真实验证 = 里程碑 7 用 Claude Code 连接跑闭环**(配 `.mcp.json` + 在设置页生成的 API Key)。

**里程碑 7 联调要点:**
1. 造一条 `confirmed` 需求 + 几篇 wiki 规范(用前端界面)。
2. 设置页生成 API Key,写 Claude Code 的 `.mcp.json`(Streamable HTTP 指向 `http://localhost:3001/mcp`,带 Authorization)。
3. 在一个空仓库让 Claude Code 走:`get_requirement_detail → search_knowledge → create_dev_plan → 逐节点 update_dev_plan_node`;网页刷新看进度树变化。
4. 权限验证:用无 `development` 职能的 Key 调 `create_dev_plan` 应被后端拒绝。

**开发-验证循环(重要):**
- 本平台代码大量由 AI 手写、依赖容器重启来编译验证。写完一批 → `docker compose restart backend`(或前端热重载)→ 看日志/界面 → 修。
