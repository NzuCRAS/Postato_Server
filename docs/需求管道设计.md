# 模块 A：需求澄清与结构化 详细设计说明

## 1. 目标
将老板或需求方提供的“模糊描述 + HTML 静态 demo”转化为一份机器可读、人类可预览的**结构化需求文档**，作为后续所有开发、测试、AI 辅助的单一事实来源。

## 2. 用户角色与核心路径
- **老板/PM**：提交原始需求素材（文本描述 + HTML 文件），发起澄清流程，确认最终需求。
- **技术负责人（你，初期）**：在 Agent 未上线时，手动与老板沟通，在平台上编辑结构化需求内容，直至需求清晰，标记为“已确认”。
- **开发人员**：通过平台预览需求和 HTML demo，并通过 MCP 在 Claude Code 中自动获取结构化需求数据。

## 3. 页面设计

### 3.1 需求创建页
路由：`/requirements/new`

布局：单栏居中卡片式，简化提交过程。
- **标题**：文本输入框。
- **描述**：Markdown 编辑器（如 `@uiw/react-md-editor`），支持粘贴图文。
- **HTML 源文件**：两种输入方式二选一
  - 上传 `.html` 文件（拖拽或点击上传）
  - 代码粘贴区（`textarea`，手动粘贴 HTML 内容）
- **操作按钮**：
  - `保存草稿` → 状态 `draft`
  - `提交并开始澄清` → 状态 `clarifying`（初期通知你来处理，未来唤醒 Agent）

### 3.2 需求详情/编辑页
路由：`/requirements/:id`

布局：左右两栏（3:5），左侧编辑，右侧预览。
#### 左侧（编辑区，仅编辑者可见）
- **基本信息**：标题、Markdown 描述（可编辑）。
- **源文件管理**：查看/替换已上传的 HTML，或修改粘贴的代码。
- **结构化需求编辑器**（核心）：
  - 初期：一个 **JSON 编辑器**（带语法高亮和实时校验），直接编辑 `structured` 对象。旁边提供 `模版插入` 按钮快速生成用户故事、模块等框架。
  - 中期（可选）：图形化表单，按模块卡片编辑，每个模块可增删验收条件。
- **操作按钮**：`保存修改`、`标记为已确认`（仅当状态为 `clarifying` 时可点，变成 `confirmed`）。

#### 右侧（预览区，只读）
- **HTML 沙箱**：
  - 使用 `<iframe sandbox="allow-scripts allow-same-origin" srcdoc={htmlCode}>` 渲染。
  - 若无 HTML 源，则显示“未提供原型”。
- **结构化需求文档视图**：
  - 根据 `structured` JSON 动态渲染美观的文档页面。
  - 渲染规则：
    - 用户故事 → 有序列表
    - 模块 → 卡片，内含名称、描述、验收条件（复选框样式）、UI 状态标签
    - 交互流程 → 箭头步骤条
    - 待澄清问题 → 警告框列表（初期为空，未来由 Agent 填充）
- 该区域所有开发者均可见，不可编辑。

### 3.3 未来 Clarify Agent 对话面板（折叠态）
- 在需求详情页右下角悬浮按钮“AI 澄清”。
- 点击弹出聊天窗口，老板可与 Agent 实时对话。
- Agent 消息可附带快速选项（单选/多选），老板点击即可回答。
- 每次对话后，右侧文档预览自动刷新，反映最新的结构化数据。

## 4. 需求状态流转
```
draft → clarifying → confirmed
  ↑________________________↓ (需要重新澄清时可手动重置)
```
- **draft**：仅创建者和管理员可见，可任意编辑。
- **clarifying**：等待澄清（初期人工、未来 Agent）。编辑权限开放给技术负责人（你）。
- **confirmed**：锁定，MCP 可读取，开发人员以此为准。小修改允许（但记录版本历史），重大变更需重新开启 clarifying。

## 5. 数据模型（核心字段）
基于 MongoDB 文档设计，若用关系库可将 `structured` 存为 JSONB。

```javascript
{
  _id: ObjectId,
  projectId: ObjectId,
  title: String,
  descriptionMd: String,          // 原始的 Markdown 描述
  sourceHtml: String,             // 文件存储路径或直接存 HTML 文本（小于1MB可存库）
  status: String,                 // 'draft' | 'clarifying' | 'confirmed' | 'deprecated'
  version: Number,                // 每次更新 structured 时自增
  structured: {                   // 机器可读核心
    user_stories: [String],
    modules: [{
      name: String,
      description: String,
      acceptance_criteria: [String],
      ui_states: [String],
      related_assets: [String]    // 可指代 HTML 内锚点或截图
    }],
    interaction_flow: String,     // 文本描述，可扩展为流程图 JSON
    ambiguous_points: [String]    // 未来由 Agent 生成
  },
  clarifyLog: [{                  // 未来对话日志
    timestamp: Date,
    role: String,                 // 'agent' | 'user'
    content: String,
    options: [String]             // 若有选择题
  }],
  createdBy: ObjectId,            // 用户 ID
  createdAt: Date,
  updatedAt: Date
}
```

## 6. API 设计
Base URL: `/api/v1`

| 方法 | 端点 | 描述 |
|------|------|------|
| POST | `/requirements` | 创建需求，接收 multipart/form-data 或 JSON（包含 `title`, `descriptionMd`, `htmlCode` 或 `htmlFile`） |
| GET | `/requirements/:id` | 获取完整需求对象（含 `structured`），需鉴权 |
| PUT | `/requirements/:id` | 部分更新（`title`, `descriptionMd`, `sourceHtml`, `structured` 等），触发 `version++` |
| PATCH | `/requirements/:id/status` | 更新状态，Body: `{ "status": "confirmed" }` |
| GET | `/requirements/:id/source-html` | 返回原始 HTML 内容（`text/html`），用于 iframe 渲染或 MCP 获取 |

初期暂不实现版本历史，通过 `version` 字段预留。

## 7. 技术实现要点
- **Markdown 编辑器**：`@uiw/react-md-editor`，支持预览和全屏。
- **JSON 编辑器**：`react-ace` 或 `@monaco-editor/react`，绑定 `structured` 对象，设置语言为 `json`。
- **HTML 沙箱**：`iframe` 的 `srcdoc` 属性，设置 `sandbox="allow-scripts allow-same-origin"`。若 HTML 引用了外部 CDN，放行默认安全策略（内部平台可接受）。
- **文件存储**：HTML 文件可存本地服务器的 `/uploads` 目录，或云存储。若文本量不大，也可直接存数据库（但考虑索引和备份，文件系统更优）。
- **状态转换**：后端中间件校验权限，仅创建者和管理员可变更状态；未来可加入角色权限矩阵。
