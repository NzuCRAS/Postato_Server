# 项目结构树「逐层共建」协议 + 工具改造 + Potato 树重做 — 设计 spec

> 状态:设计已对齐(2026-06-11),待写实施计划。承接 `2026-06-11-arch-upsert-bootstrap-design.md`(那轮补了整树写入能力并首次自举,但树质量不足)。

## 背景与问题
首次自举录入的 Potato 结构树暴露三个问题:
1. **层级语义错位**:把 `DevPlanService` 等**实现单元**直接放进 L3,而方法论里 L3=业务模块/聚合、L4=实现单元(`docs/项目树形架构设计.md`)。
2. **说明含糊**:`description` 多为同义反复,讲不清职责与边界。
3. **划分粗、不够深**:每个 L2 上下文只挂 1 个 service,没有真正的业务模块层。

**根因**:一次性 AI 生成(`upsert_architecture` 灌整树),没有逐层校准语义。**结论**:建树要改成「渐进、逐层(BFS)、对话澄清」的共建,并从**工具形态**与**协议**两端共同保障。

## 已对齐的决策
- **形态**:AI 经 MCP 的共建,对话发生在 Claude Code 会话(非平台网页 Agent —— 后者列为后续独立切片)。
- **范围**:定协议 + 用它**重做 Potato 自己这棵树**(作为协议第一个示范)。
- **深度**:手动共建到 **L3 业务模块**;L4 实现单元交 `.project.yaml` + `sync_project_modules`。
- **约束分工**:**工具硬化逐层节奏** + **协议软保障澄清质量**。
- **后端能力**:`upsertTree`/`upsert-tree` 端点**保留**为内部通用能力,仅在 MCP 契约层收窄(AI 只见逐层工具)。

## 一、工具改造(硬约束:物理上只能逐层)
- **废弃 MCP `upsert_architecture`**(整树形态)。
- **新增 MCP `upsert_arch_layer(project_id, parent_path, nodes[])`**:一次建/更新 `parent_path` **下的一层**兄弟节点;**schema 不含 `children`**(嵌套无法表达 → 物理强制逐层)。仍是 upsert 语义(同层 path 命中则幂等更新)。
- **后端零改动**:新工具复用现有 `POST /arch/upsert-tree`,传单层、无 `children` 的 `nodes`;`upsertTree` 本就能处理(只建一层)。
- `index.ts`:`TOOL_COUNT` 维持 14(删一加一),注释把 `upsert_architecture` 换成 `upsert_arch_layer`。
- 验证:`tsc --noEmit` + 重启 `mcp-server`。

## 二、逐层共建协议(软约束:澄清质量)
落地三处:
1. **知识库 `/agent/arch-coauthoring`** —— 协议全文,AI 可 `search_knowledge` 检索。
2. **CLAUDE.md** 新增「项目结构树共建」小节 —— 触发规则 + 指针(硬规定:建树须走逐层共建协议、禁止一次性生成),指向知识库全文。
3. **`upsert_arch_layer` 工具描述** —— 补一句「配合逐层共建协议,先澄清当层再写入」。

### 2.1 流程(严格 BFS)
- **触发**:用户表达「建项目 / 理结构 / 搭结构树」意图时,AI **主动发起共建**(而非直接生成)。
- **第 0 步 — 目标对齐**:先问清这棵树服务谁、回答什么问题(可定位/可追溯/可度量)、项目边界;`get_project_detail` + `get_architecture` 读背景与现状(支持断点续建)。
- **逐层推进**(同层全部澄清完再下沉):
  - **L0 系统** → 确认系统名 + 一句话定位。
  - **L1 领域** → 对话「有哪些核心业务能力域?哪些是支撑域?」,逐个对齐职责。
  - **L2 限界上下文** → 对**每个** L1:「有哪些边界清晰、可独立演进的上下文?」
  - **L3 业务模块** → 对**每个** L2:「有哪些独立职责的业务模块?」止于 L3。
- **每层产出与写入**:AI 把当层结构化(title/type/tags/description/related_code/related_docs)→ 呈现给用户确认 → 确认后 `upsert_arch_layer`(`parent_path` 指向上层)**增量写当层**。
- **中断与恢复**:每层即时落库;`get_architecture` 读进度,从未完成层续建。
- **停止**:到 L3,或用户喊停。

### 2.2 层级语义铁律
| 层 | 是什么 | 禁止 |
|---|---|---|
| L0 系统 | 可独立交付的产品/平台 | — |
| L1 领域 | 核心业务能力聚类 | ❌ 前端/后端等技术分层冒充领域 |
| L2 限界上下文 | 边界清晰、可独立演进的业务单元 | — |
| L3 业务模块 | 上下文内独立职责的业务单元(如「工作日志模块」) | ❌ `XxxService` 等实现单元 |
| L4 实现单元(交 sync) | 具体 Service/类/页面 | — |

### 2.3 节点质量标准
- `description` 必须答出 **职责(负责什么)+ 边界(不负责什么 / 与同层兄弟的区别)**,**禁止同义反复**(❌「认证上下文:负责认证」)。
- `type` 用受控词表 `system/domain/context/module`;`tags` 给跨切面维度(安全/基础设施…);`related_code` 用 glob;`related_docs` 挂知识库 path。

### 2.4 每层提问模板(示例)
- **L1**:「系统的核心业务能力有哪些?哪些属支撑性(集成/基础设施)?」
- **L2**:「X 领域里有哪些边界清晰的上下文?认证与权限是一个还是两个?」
- **L3**:「X 上下文里有哪些独立职责的业务模块?」

## 三、重做 Potato 树(协议首个示范)
- **不在本 spec 预先固定新结构**(否则又回到「一次性生成」);具体 L1–L3 在实施阶段**按协议逐层对话共建**得出。
- **起点参考**(实施时逐层确认、可增删):现有 6 个 L1 领域(用户/需求/研发过程/知识/项目/集成)划分大体合理,L1/L2 可在共建中校准保留;**重点重构 L3**。
- **现有错位节点处理**:现有 L3 的 `XxxService`(实现单元)用现有 `archive`(子树归档)**归档**,换成逐层共建得出的真正业务模块;`JWT 实现`(L3·sync)等实现单元按新语义属 L4,本轮不手动建 L4 → 实施时决定归档或留作 L4·sync 示例。

## 六、非目标 / 风险
- **非目标**:平台网页向导 Agent(后续切片);L4 手动维护(交 sync);移除后端整树能力(保留)。
- **风险**:协议是软约束(质量层),依赖 AI 遵守 —— 但节奏已由工具硬化,且与现有 Herness 契约同机制(已验证)。废弃 `upsert_architecture` 是上一轮(`461b6d7`)刚加的工具,属设计演进,诚实记录。`create_requirement` 需 product 职能。
